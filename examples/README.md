# Example: Magic-Link Login Feature

This directory contains a **complete, end-to-end example** of every SDD artifact for a single
realistic feature: passwordless authentication via magic link.

Use it as:

- A reference when you're filling in your own templates and want to see what "done" looks like.
- A starter you can copy into a new project and adapt.
- Material for onboarding teammates to SDD.

## Layout

This example mirrors the structure SDD expects at the root of a real project:

```text
examples/
├── README.md              ← you are here
├── constitution.md        ← Phase 0 — project-level immutable constraints
└── specs/
    └── magic-link-login/  ← one feature branch
        ├── spec.md          (Phase 1 — requirements, MoSCoW ACs, boundaries)
        ├── plan.md          (Phase 2 — architecture, risks, AC coverage map)
        ├── data-model.md    (Phase 2 — entities, indexes, migrations)
        ├── contracts/
        │   └── auth-api.md  (Phase 2 — locked API contract)
        └── tasks.md         (Phase 3 — atomic, test-first, dependency-mapped)
```

## How this example was built

The example was generated by following the workflow exactly as the skill describes:

1. **Phase 0 — Constitution.** Drafted a minimal but realistic `constitution.md` for a
   TypeScript/Postgres stack (see `constitution.md`).
2. **Phase 1 — Specify.** Wrote `spec.md` with 4 user stories, 6 acceptance criteria
   (mix of MUST/SHOULD/WONT), an explicit Boundaries block, and 3 error-case ACs.
3. **Clarify.** Resolved every `[NEEDS CLARIFICATION]` in `spec.md`; final document has
   none remaining.
4. **Phase 2 — Plan.** Designed 4 components, an AC coverage map covering every MUST AC,
   and a risk table with 3 entries. Locked `contracts/auth-api.md` after this phase.
5. **Phase 3 — Tasks.** Broke implementation into 11 atomic tasks (3 test tasks before
   3 implementation tasks, all dependency-mapped, `[P]` for parallelizable).
6. **Phase 4 — Implement.** *(Out of scope for this docs-only repository — but each task
   in `tasks.md` is ready to feed into a fresh AI session with the prompt pattern from
   `references/prompt-patterns.md`.)*

## Caveats

This is an example, not production code. The example deliberately:

- Uses placeholder file paths (real projects substitute their own).
- Omits the `decision_log.md` and `research.md` artifacts (both are optional — see
  `references/artifact-templates.md`).
- Skips Phase 4 (implementation) and Phase 5 (validation) since this repository has no
  application code to validate against.
